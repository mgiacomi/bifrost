# Research: ENG-024 output schema support for YAML skills

**Date**: 2026-03-23T23:22:30.8361635-07:00

## Research Question

What exists in the codebase today for YAML-skill output schemas, and which current components own YAML manifest parsing, validation, advisor wiring, and execution behavior relevant to `output_schema` support?

## Summary

The current codebase has a typed YAML-skill manifest model, boot-time YAML discovery, strict unknown-field rejection, model and thinking-level validation, optional regex linter validation, and ChatClient advisor wiring for that linter. The manifest model does not currently include `output_schema` or `output_schema_max_retries`, and the starter source tree contains no `output_schema`, `OutputSchema`, or `BifrostOutputSchemaValidationException` references from this research pass.

Today, structured validation for YAML skills is limited to the existing regex linter path. That path is represented in the manifest, validated in `YamlSkillCatalog`, converted into a `LinterCallAdvisor` by `DefaultSkillAdvisorResolver`, and attached to skill-specific `ChatClient` instances by `SpringAiSkillChatClientFactory`. Mission execution currently sends a system prompt plus the user objective and returns the model text content without any JSON parsing or schema-validation stage.

### Advisor Wiring And Current Output Validation

`DefaultSkillAdvisorResolver` is the component that turns manifest-level YAML settings into Spring AI advisors. In the current code, it only inspects `definition.linter()`, and only produces an advisor when the linter type is `regex` and a regex block is present ([DefaultSkillAdvisorResolver.java](/C:/opendev/code/bifrost/bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/chat/DefaultSkillAdvisorResolver.java#L24), [DefaultSkillAdvisorResolver.java](/C:/opendev/code/bifrost/bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/chat/DefaultSkillAdvisorResolver.java#L27), [DefaultSkillAdvisorResolver.java](/C:/opendev/code/bifrost/bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/chat/DefaultSkillAdvisorResolver.java#L31)).

When a regex linter is present, the resolver builds `LinterCallAdvisor` with the skill name, linter type, compiled pattern, message text, max retries, and a callback that records linter outcomes on the current Bifrost session via `ExecutionStateService` ([DefaultSkillAdvisorResolver.java](/C:/opendev/code/bifrost/bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/chat/DefaultSkillAdvisorResolver.java#L34), [DefaultSkillAdvisorResolver.java](/C:/opendev/code/bifrost/bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/chat/DefaultSkillAdvisorResolver.java#L38)).

`LinterCallAdvisor` implements the current retry-and-hint output validation loop. It calls the model, extracts assistant text, matches it against the configured regex, records `PASSED`, `RETRYING`, or `EXHAUSTED` outcomes, and appends a system-message hint before retrying when the regex does not match ([LinterCallAdvisor.java](/C:/opendev/code/bifrost/bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/linter/LinterCallAdvisor.java#L18), [LinterCallAdvisor.java](/C:/opendev/code/bifrost/bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/linter/LinterCallAdvisor.java#L49), [LinterCallAdvisor.java](/C:/opendev/code/bifrost/bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/linter/LinterCallAdvisor.java#L57), [LinterCallAdvisor.java](/C:/opendev/code/bifrost/bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/linter/LinterCallAdvisor.java#L63), [LinterCallAdvisor.java](/C:/opendev/code/bifrost/bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/linter/LinterCallAdvisor.java#L67), [LinterCallAdvisor.java](/C:/opendev/code/bifrost/bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/linter/LinterCallAdvisor.java#L105)).

There is no parallel advisor or validator in the current codebase for JSON parsing, output-schema checking, issue accumulation, or output-schema-specific terminal exceptions from this research pass.

## Architecture Documentation

The current YAML-skill architecture is layered as follows:

1. YAML manifests are discovered and deserialized by `YamlSkillCatalog`.
2. The catalog validates the currently supported manifest contract and resolves effective execution settings.
3. `YamlSkillDefinition` carries the manifest plus resolved execution configuration into the runtime.
4. `YamlSkillCapabilityRegistrar` registers YAML skills into the capability registry and links mapped skills to discovered deterministic targets.
5. `DefaultSkillAdvisorResolver` translates manifest-level verification settings into advisors.
6. `SpringAiSkillChatClientFactory` creates a skill-specific `ChatClient` with provider-specific chat options and any resolved advisors.
7. `ExecutionCoordinator` and `DefaultMissionExecutionEngine` execute the top-level skill by building a prompt, calling the model, and returning the resulting text.

Within that architecture, the only implemented post-generation validation loop for YAML skills is the regex linter advisor path.
