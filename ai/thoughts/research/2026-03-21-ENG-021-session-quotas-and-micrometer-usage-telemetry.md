## Summary
The current Bifrost runtime already has two session-level guardrails in the normal execution path: stack-depth enforcement on `BifrostSession.pushFrame(...)` and mission timeout enforcement in `DefaultMissionExecutionEngine.executeMission(...)` ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/core/BifrostSession.java:207](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/core/BifrostSession.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/DefaultMissionExecutionEngine.java:72](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/DefaultMissionExecutionEngine.java)).

Runtime state changes already flow through a dedicated `ExecutionStateService` abstraction, which records plan creation/updates, tool calls/results, linter outcomes, and errors into the session journal ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/state/DefaultExecutionStateService.java:25](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/state/DefaultExecutionStateService.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/core/BifrostSession.java:90](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/core/BifrostSession.java)). Tool execution is logged through `DefaultToolCallbackFactory`, and YAML lint retries are driven by `DefaultSkillAdvisorResolver` plus `LinterCallAdvisor` ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java:59](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/chat/DefaultSkillAdvisorResolver.java:24](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/chat/DefaultSkillAdvisorResolver.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/linter/LinterCallAdvisor.java:49](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/linter/LinterCallAdvisor.java)).

Within the checked-in source tree, there is currently no Micrometer-specific source wiring and no session-quota abstraction or configuration beyond `maxDepth` and `missionTimeout`. The starter POM declares Spring Boot, Spring AI, and validation/security/test dependencies, but no explicit Micrometer dependency is declared in the module POM ([bifrost-spring-boot-starter/pom.xml:18](../../../bifrost-spring-boot-starter/pom.xml)). A search across `bifrost-spring-boot-starter/src` for `MeterRegistry`, `io.micrometer`, and `micrometer` returned no matches at research time.

## Detailed Findings

### Session guardrails currently implemented
- `BifrostSessionProperties` exposes two `bifrost.session` settings: `maxDepth` and `missionTimeout`, with defaults of `32` and `60s` ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/autoconfigure/BifrostSessionProperties.java:11](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/autoconfigure/BifrostSessionProperties.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/autoconfigure/BifrostSessionProperties.java:14](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/autoconfigure/BifrostSessionProperties.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/autoconfigure/BifrostSessionProperties.java:20](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/autoconfigure/BifrostSessionProperties.java)).
- `BifrostAutoConfiguration` wires `BifrostSessionRunner` from `maxDepth`, constructs a virtual-thread mission executor, and passes `missionTimeout` into `DefaultMissionExecutionEngine` ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/autoconfigure/BifrostAutoConfiguration.java:87](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/autoconfigure/BifrostAutoConfiguration.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/autoconfigure/BifrostAutoConfiguration.java:196](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/autoconfigure/BifrostAutoConfiguration.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/autoconfigure/BifrostAutoConfiguration.java:206](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/autoconfigure/BifrostAutoConfiguration.java)).
- `BifrostSession.pushFrame(...)` enforces stack depth by comparing current frame count to `maxDepth` and throwing `BifrostStackOverflowException` when the limit is reached ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/core/BifrostSession.java:207](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/core/BifrostSession.java)).
- `ExecutionCoordinator.execute(...)` opens the mission frame before invoking the mission engine and always closes it in a `finally` block, so the depth guard runs in the main execution path for skill execution ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/core/ExecutionCoordinator.java:52](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/core/ExecutionCoordinator.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/core/ExecutionCoordinator.java:80](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/core/ExecutionCoordinator.java)).
- `DefaultMissionExecutionEngine.executeMission(...)` submits mission work to the configured executor and uses `Future.get(timeout)` with cancellation on timeout or interruption, converting those failures into `BifrostMissionTimeoutException` ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/DefaultMissionExecutionEngine.java:57](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/DefaultMissionExecutionEngine.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/DefaultMissionExecutionEngine.java:72](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/DefaultMissionExecutionEngine.java)).

### Tool execution flow and what it currently records
- `DefaultToolCallbackFactory` wraps each visible capability in a `FunctionToolCallback` and routes invocation through `invokeCapability(...)` ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java:35](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java:46](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java)).
- At tool start, it asks `PlanningService` to link the call to a plan task. It then logs either a linked `TOOL_CALL` or an `UNPLANNED_TOOL_EXECUTION` journal event depending on whether a unique ready task was matched ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java:63](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java:66](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java)).
- On successful tool completion, it updates the plan state and logs a `TOOL_RESULT` event with the returned result payload ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java:79](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java:84](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java)).
- On tool failure, it marks the task failed in the plan when a linked task exists and writes an `ERROR` journal payload containing tool name, linked task ID, exception message, and exception type ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java:89](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java:93](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java)).

### Linter flow and currently available linter telemetry
- `DefaultSkillAdvisorResolver` inspects YAML skill definitions for a `regex` linter and, when present, installs a `LinterCallAdvisor` whose outcome recorder writes into the current Bifrost session via `ExecutionStateService.recordLinterOutcome(...)` ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/chat/DefaultSkillAdvisorResolver.java:25](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/chat/DefaultSkillAdvisorResolver.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/chat/DefaultSkillAdvisorResolver.java:38](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/chat/DefaultSkillAdvisorResolver.java)).
- `LinterCallAdvisor` retries model calls until the configured regex matches or retries are exhausted. For each attempt it records a `LinterOutcome` with skill name, linter type, attempt count, retry count, max retries, status, and detail ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/linter/LinterCallAdvisor.java:54](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/linter/LinterCallAdvisor.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/linter/LinterCallAdvisor.java:85](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/linter/LinterCallAdvisor.java)).
- Recorded linter outcomes are added both to the mutated `ChatClientResponse` context and to the session journal through `DefaultExecutionStateService` and `BifrostSession.logLinterOutcome(...)` ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/linter/LinterCallAdvisor.java:89](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/linter/LinterCallAdvisor.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/state/DefaultExecutionStateService.java:117](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/state/DefaultExecutionStateService.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/core/BifrostSession.java:114](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/core/BifrostSession.java)).

## Architecture Documentation
The current runtime is organized around a session object plus service boundaries. `ExecutionCoordinator` is the orchestration entry point for YAML skills: it resolves the YAML definition and capability metadata, applies access checks, clears any prior plan, opens the root mission frame, creates the skill-specific chat client, exposes visible tools, and delegates mission execution to `MissionExecutionEngine` ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/core/ExecutionCoordinator.java:52](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/core/ExecutionCoordinator.java)).

Session mutations are centralized in `DefaultExecutionStateService`, which updates `BifrostSession` rather than having each caller write directly to journal and plan structures. Tool callbacks and linter advisors both feed into this boundary, so current runtime telemetry-like data is session-local journal data rather than an external metrics registry ([bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/state/DefaultExecutionStateService.java:17](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/state/DefaultExecutionStateService.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java:19](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/runtime/tool/DefaultToolCallbackFactory.java), [bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/chat/DefaultSkillAdvisorResolver.java:14](../../../bifrost-spring-boot-starter/src/main/java/com/lokiscale/bifrost/chat/DefaultSkillAdvisorResolver.java)).

Planning is optional per skill. When enabled, the model first returns an `ExecutionPlan`, which is then used to shape the execution prompt and to associate later tool calls with plan tasks. Current guardrail enforcement happens at frame push time and mission future timeout time; current runtime accounting happens through journal events and the `lastLinterOutcome` snapshot in the session object.

## Follow-up Research 2026-03-21T17:02:00-07:00

### Spring AI usage metadata surfaces in the version used by this repo
- The current repo depends on Spring AI `1.1.2` modules through the starter POM ([bifrost-spring-boot-starter/pom.xml:23](../../../bifrost-spring-boot-starter/pom.xml), [bifrost-spring-boot-starter/pom.xml:31](../../../bifrost-spring-boot-starter/pom.xml)).
- In Spring AI `spring-ai-model-1.1.2-sources.jar`, `org/springframework/ai/chat/model/ChatResponse.java` defines `getMetadata()` and carries a `ChatResponseMetadata` object. The same source shows the builder copying `usage` metadata when a response is rebuilt.
- In `spring-ai-model-1.1.2-sources.jar`, `org/springframework/ai/chat/metadata/ChatResponseMetadata.java` defines a `Usage usage` field and exposes it through `getUsage()`. In the same module, `org/springframework/ai/chat/metadata/Usage.java` defines `getPromptTokens()`, `getCompletionTokens()`, `getTotalTokens()`, and `getNativeUsage()`, while `DefaultUsage` supplies the default implementation.
- This means the common Spring AI surface available to Bifrost in the current dependency set is `ChatResponse.getMetadata().getUsage()`, with provider-specific details available via `Usage.getNativeUsage()`.

### Provider-specific usage metadata exposed by Spring AI 1.1.2
- OpenAI: `spring-ai-openai-1.1.2-sources.jar` shows `org/springframework/ai/openai/OpenAiChatModel.java` reading `chatCompletion.usage()`, converting it with `getDefaultUsage(...)`, and accumulating it with `UsageCalculator.getCumulativeUsage(...)`. The matching `org/springframework/ai/openai/api/OpenAiApi.java` source defines an `OpenAiApi.Usage` record with `completion_tokens`, `prompt_tokens`, `total_tokens`, and nested prompt/completion token details including `reasoning_tokens`.
- OpenAI streaming: the same `OpenAiChatModel` source includes logic around `request.streamOptions().includeUsage()`, and comments indicate usage is present only in the final streaming response when that option is enabled.
- Anthropic: `spring-ai-anthropic-1.1.2-sources.jar` shows `org/springframework/ai/anthropic/AnthropicChatModel.java` reading `completionResponse.usage()`, mapping it to `DefaultUsage`, and accumulating it cumulatively. `org/springframework/ai/anthropic/api/AnthropicApi.java` defines a `Usage` record with `input_tokens`, `output_tokens`, `cache_creation_input_tokens`, and `cache_read_input_tokens`.
- Anthropic thinking content: the same `AnthropicChatModel` source handles `THINKING` and `REDACTED_THINKING` content blocks in response generation, so the provider adapter distinguishes those response forms separately from token usage metadata.
- Google GenAI: `spring-ai-google-genai-1.1.2-sources.jar` shows `org/springframework/ai/google/genai/GoogleGenAiChatModel.java` reading `generateContentResponse.usageMetadata()`, then either constructing default usage or `GoogleGenAiUsage.from(...)` depending on the `includeExtendedUsageMetadata` option. `org/springframework/ai/google/genai/metadata/GoogleGenAiUsage.java` extends `DefaultUsage` with `thoughtsTokenCount`, `cachedContentTokenCount`, `toolUsePromptTokenCount`, modality breakdowns, traffic type, and the native `GenerateContentResponseUsageMetadata`.
- Ollama: `spring-ai-ollama-1.1.2-sources.jar` shows `org/springframework/ai/ollama/OllamaChatModel.java` building `DefaultUsage` from `promptEvalCount` and `evalCount`, then storing those counts plus durations in `ChatResponseMetadata`. The matching `org/springframework/ai/ollama/api/OllamaApi.java` source defines those fields on `ChatResponse`.

## Implementation Alignment Note 2026-03-21T17:31:00-07:00

- ENG-021 was implemented by extending `bifrost.session` with `quotas.*` settings for skill invocations, tool invocations, linter retries, model calls, and usage units, then storing cumulative usage in a dedicated `SessionUsageSnapshot` on `BifrostSession`.
- The runtime now records mission starts, model responses, tool calls, and linter retry outcomes through `DefaultSessionUsageService`, which throws `BifrostQuotaExceededException` with `GuardrailType`, limit, and observed values when a quota is exceeded.
- `DefaultMissionExecutionEngine` now prefers `ChatClient.CallResponseSpec.chatResponse()` so it can inspect `ChatResponseMetadata.getUsage()`, but still falls back to `.content()` for clients that do not expose `ChatResponse`.
- Exact model usage is recorded from Spring AI `Usage.getPromptTokens()`, `getCompletionTokens()`, and `getTotalTokens()` when present; otherwise Bifrost falls back to a lightweight deterministic heuristic based on prompt/response text length.
- Micrometer telemetry is now emitted through `UsageMetricsRecorder` with the implemented meter names `bifrost.skill.calls`, `bifrost.model.calls`, `bifrost.model.prompt.units`, `bifrost.model.completion.units`, `bifrost.model.usage.units`, `bifrost.tool.calls`, `bifrost.linter.outcomes`, and `bifrost.guardrail.trips`.
