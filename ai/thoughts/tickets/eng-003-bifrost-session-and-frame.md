# Ticket: eng-003-bifrost-session-and-frame.md
## Issue: Implement BifrostSession and ExecutionFrame Lifecycle

### Context
Bifrost sessions hold the thread-local or scoped session state during LLM operation. Virtual threads are in use, so concurrency mechanisms need to remain virtual-thread friendly.

### Requirements
1. **`BifrostSession` Context:**
   - Implement `BifrostSession`. This must live for the duration of a specific task or mission.
   - Add a unique session ID, owned and generated by `BifrostSession` itself by default.
   - Include a `ReentrantLock` for state modifications; avoid `synchronized` blocks to prevent pinning virtual-thread carrier threads during IO operations.
   - `BifrostSession.getCurrentSession()` should fail clearly when no scoped session is active rather than returning `null`.
2. **`ExecutionFrame` and Stack Tracking:**
   - Create an `ExecutionFrame` POJO tracking operation kind, routed target, immutable input parameters, and enough correlation metadata to support future nested execution work.
   - `BifrostSession` pushes and pops `ExecutionFrame`s as a stack for recursion tracking.
   - Enforce a configurable `MAX_DEPTH` to prevent deep unconstrained recursion or loops. Throw `BifrostStackOverflowException` if the limit is exceeded.
   - Popping an empty frame stack should fail fast rather than silently returning `null`.
3. **Thread Scoping Strategy:**
   - Use a `ThreadLocal`-backed session holder with strict `try/finally` cleanup so `BifrostSession.getCurrentSession()` can be resolved statically and the session can also be injected safely per execution flow without passing it through all method signatures.
   - Restore any previously-bound session when nested execution boundaries complete so session state does not leak across flows.
4. **Configuration:**
   - Provide a sensible default `MAX_DEPTH`.
   - Allow `MAX_DEPTH` to be configured via `application.yml`.

### Acceptance Criteria
- `BifrostSession` uses `ReentrantLock` for thread-safe mutations.
- `ExecutionFrame` push/pop is implemented with tested `MAX_DEPTH` fail-fast circuit breakers.
- A static or injected access pattern correctly resolves the session boundaries in unit testing across multiple concurrent virtual threads.
- `BifrostSession.getCurrentSession()` and empty-stack pops fail clearly outside valid execution boundaries.
- `MAX_DEPTH` has a starter default and can be overridden via Spring Boot configuration.
